# FloatingAccessoryKit v0.3.0 API redesign

Status: implemented and verified for v0.3.0.

This document is the source of truth for the v0.3.0 API and ownership migration.

## 0. Scope contract

### Outcome

- Make each `UITabBarController` the unambiguous lifetime owner of exactly one floating-accessory controller.
- Make content, placement, visibility, measurement, and removal independent contracts so one update cannot silently reset another.
- Preserve native `UITabAccessory` presentation on iOS 26 and the overlay presentation on iOS 18 while giving both paths the same public semantics.
- Verify the shipped product through a separate-package consumer, not only through `@testable` tests.

### Compatibility policy

This is a source-breaking redesign for v0.3.0. The repository's Sample app and README will migrate in the same branch; no deprecated compatibility shim will remain. README will gain `Migration > v0.3.0`, using the versioned format requested for this release.

The existing deployment contract remains iOS 18.0+, iPhone only, Swift tools 6.2, and Swift 6 language mode. This is an explicit exception to the newer general design baseline because lowering support was already a published package decision and changing it is unrelated to the audited invariants.

Because the product is a reusable public package and no second external app is in this repository, a separate Swift package fixture will act as the proxy second consumer. It will import and link the library product without `@testable` or package-level access.

### Non-goals

- Adding iPad, macOS, SwiftUI, or non-tab-bar container support.
- Removing the iOS 26 native presentation or its documented reliance on undocumented layout behavior.
- Changing the package, product, target, deployment target, or distribution model.
- Preserving source compatibility with v0.2.x.

## 1. Current measurements and findings

### Baseline

- Package graph: one library product, one implementation target, one test target, no external dependencies.
- Implementation imports: UIKit throughout; Objective-C runtime in container sizing, hit testing, and lifecycle hooks. No `@_exported import`.
- Access distribution: 6 `public` declarations, 0 `open`, 0 `package` in `Sources/FloatingAccessoryKit`.
- Platform gates: 0 `#if canImport` / `#if os`; the iOS 26 choice is made in `TabBarAccessoryController`, with native types marked `@available(iOS 26.0, *)`.
- Implementation size: 2,106 lines. Largest files are `OverlayTabBarAccessoryRenderer.swift` at 805 lines, `NativeAccessoryContainerLayout.swift` at 376, and `NativeTabBarAccessoryRenderer.swift` at 343.
- State concentration: `OverlayTabBarAccessoryRenderer` has 18 top-level stored properties and owns semantic state, render artifacts, safe-area composition, tab-bar visibility, measurement constraints, and animation generation. The native coordinator has 13 top-level stored properties and independently mirrors content, position, and visibility.

### Implemented result

- Package graph remains one library product, one implementation target, one test target, and no external dependencies.
- Public surface has 15 `public` declaration lines, 0 `open`, and 0 `package`; every public operation is exercised by the separate-package fixture.
- Implementation size is 3,137 lines. The largest files are `OverlayTabBarAccessoryRenderer.swift` at 718 lines, `NativeTabBarAccessoryRenderer.swift` at 485, and `NativeAccessoryContainerLayout.swift` at 397.
- `OverlayTabBarAccessoryRenderer` now has 14 top-level stored properties and `NativeTabBarAccessoryRenderer` has 10. Neither renderer stores semantic content, placement, or requested visibility.
- Host observation is local to one controller: root geometry uses an inert observation view, while `TabBarAccessoryHostObservation` owns per-host selection inputs and the iOS 18 overlay's selected-child safe area, tab-bar frame, appearance, visibility, and button-geometry inputs. No process-global lifecycle swizzle or delegate interception remains.
- Owners that mutate consumer objects restore those mutations on explicit removal and on host deallocation: `AccessoryContentHostView` restores the content view's autoresizing setting, and `OverlaySafeAreaAdjustment` restores its exact safe-area contribution when the consumer has not overwritten it.

### Findings

F1. `TabBarAccessoryController.init(tabBarController:)` permits multiple controllers for one host even though UIKit exposes one `bottomAccessory` slot. A stale controller can clear or overwrite a newer package accessory, while overlay controllers can install duplicate hosts and safe-area adjustments.

F2. `setContent(_:position:animated:)` combines install, replace, move, manual remeasurement, and removal. Both renderers also reset `isHidden` to `false`, so resubmitting the same view after an intrinsic-size change can unexpectedly show and move it.

F3. Semantic state is duplicated in platform renderers, while `TabBarAccessoryViewLifecycleHooks` keeps a process-global weak registry and swizzles four `UITabBarController` lifecycle methods. Dropping the public controller can leave installed render artifacts with no semantic owner.

F4. The overlay derives a consumer's safe-area baseline by subtracting the package's previous delta from a composite value. A consumer write while installed can therefore be misclassified and later restored to the wrong value.

F5. Animated overlay removal keeps ownership of the consumer view until animation completion. Reparenting the view immediately after `setContent(nil, animated: true)` can be undone by the old completion calling `removeFromSuperview()`.

F6. Native placement and hit-testing hooks depend on undocumented selectors and class layout. Hook-installation failure is silent, so a degraded presentation is not observable during development.

F7. README says “Use any `UIView`”, while the public DocC contract requires intrinsic sizing and forbids required width or height constraints. The implementation, documentation, and tests do not express one sizing contract.

F8. All package tests use `@testable import FloatingAccessoryKit`; the Sample is a public-only consumer but CI does not build it. The published public surface is therefore not independently compiled.

F9. Simulator measurement on iOS 26 showed that a trailing accessory's unchanged add button moved three points when a second button was inserted. The native host kept its trailing edge, but `AccessoryContentHostView` centered consumer content inside changing managed widths, so the consumer's trailing inset changed. The Sample also mutated its hierarchy before starting a second animation, splitting one Liquid Glass transition across two animation owners.

## 2. Target design

### 2.1 Target graph and owner map

The package remains one product and one implementation target: all code has the same UIKit consumer, versioning boundary, deployment constraint, and distribution unit. Splitting targets would add access and dependency ceremony without isolating a real consumer or dependency boundary.

Within the target, dependencies are one-way:

```text
UITabBarController extension
    -> TabBarAccessoryController (semantic state + public operations)
        -> AccessoryRendering (render command boundary)
            -> NativeTabBarAccessoryRenderer (iOS 26+ artifacts)
            -> OverlayTabBarAccessoryRenderer (iOS 18 artifacts)

Shared rendering owners:
    AccessoryContentHostView (content containment + measurement notifications)
    TabBarAccessoryLayoutAnimator (animation policy)

Native-only owners:
    NativeAccessoryContainerLayout
    NativeAccessoryEnvironmentObservation
    NativeAccessoryHitTesting

Overlay-only owners:
    OverlayAccessoryHostView
    OverlaySafeAreaAdjustment
    TabBarRevealHitAreaView
```

Owner map:

| Responsibility | Owner | Invariant |
| --- | --- | --- |
| Semantic content, placement, requested visibility | `TabBarAccessoryController` | One state owner per `UITabBarController`; renderers never decide or reset semantic state. |
| Controller lifetime | `UITabBarController.floatingAccessoryController` association | Repeated access returns the same controller; the host retains it for the host lifetime. |
| Platform selection | Controller composition root | Exactly one renderer is selected once from OS availability. |
| Host geometry, appearance, and selected child | `TabBarAccessoryHostObservation` | Root geometry and selection update only the owning renderer; iOS 18 additionally observes selected-child safe-area changes and every tab-bar input consumed by the overlay, without global hooks or delegate interception. |
| Content containment, alignment, and layout-driven preferred-size observation | `AccessoryContentHostView` | Consumer content has one superview; its logical edge follows the requested position while its host changes size; actual host layout triggers remeasurement without polling. |
| Explicit content update transaction | `TabBarAccessoryController` | Consumer mutations and remeasurement share one UIKit animation transaction and do not require content resubmission. |
| Native `bottomAccessory` slot | `NativeTabBarAccessoryRenderer` | It clears only the exact `UITabAccessory` instance it installed. |
| Native minimized/regular environment changes | `NativeAccessoryEnvironmentObservation` | The bound container's frame and the content host's `UITraitTabAccessoryEnvironment` invalidate native layout only while the presentation is owned. |
| Overlay safe-area contribution | `OverlaySafeAreaAdjustment` | Consumer baseline, package contribution, and last package write are stored separately; no baseline is inferred by subtraction. |
| Animation artifact | Renderer transition state | Semantic detach is synchronous; only a snapshot may outlive `removeContent` for animation. |
| Native private capability | Native helper owners | Missing selectors degrade to UIKit's public default layout, emit one diagnostic, and never remove a foreign accessory. |

Physical source layout follows these owner clusters:

```text
Sources/FloatingAccessoryKit/
  Public/
    UITabBarController+FloatingAccessory.swift
    TabBarAccessoryController.swift
  Rendering/
    AccessoryContentHostView.swift
    AccessoryRendering.swift
    FloatingAccessoryDiagnostics.swift
    TabBarAccessoryContentMeasurement.swift
    TabBarAccessoryHostObservation.swift
    TabBarAccessoryLayoutAnimator.swift
    TabBarAccessoryLayoutObservationView.swift
    TabBarAccessoryTabBarButtons.swift
  Rendering/Native/
    NativeTabBarAccessoryRenderer.swift
    NativeAccessoryContainerLayout.swift
    NativeAccessoryEnvironmentObservation.swift
    NativeAccessoryHitTesting.swift
  Rendering/Overlay/
    OverlayTabBarAccessoryRenderer.swift
    OverlayAccessoryHostView.swift
    OverlaySafeAreaAdjustment.swift
    TabBarRevealHitAreaView.swift
```

#### Considered alternatives

1. **Recommended: host-associated `TabBarAccessoryController`.** This preserves the useful controller vocabulary while making construction and lifetime impossible to misuse. It removes the consumer's retention burden and matches UIKit's single accessory slot.
2. **Public `TabBarAccessory` wrapper assigned to `UITabBarController`.** This is superficially closer to UIKit 26, but adds a second public object around a `UIView`, permits the same wrapper or view to be assigned to multiple hosts, and has no second consumer need. Rejected as a new identity and misuse surface.
3. **Keep the public initializer and precondition on duplicates.** An initializer cannot return the existing owner, so valid access still depends on call-site discipline and retention. Rejected because the illegal state remains representable.

### 2.2 Public API sketch

```swift
import UIKit

@MainActor
public extension UITabBarController {
    /// The floating-accessory controller owned by this tab bar controller.
    var floatingAccessoryController: TabBarAccessoryController { get }
}

@MainActor
public final class TabBarAccessoryController {
    public struct Position: Hashable, Sendable {
        public static let leading: Position
        public static let center: Position
        public static let trailing: Position
    }

    /// The currently installed consumer view, or `nil` when absent.
    public var contentView: UIView? { get }

    /// The current requested placement.
    public var position: Position { get }

    /// The requested visibility, retained even while content is absent.
    public var isHidden: Bool { get }

    /// Installs or replaces content and optionally changes placement.
    /// Omitting `position` preserves the current requested placement, which is
    /// initially `.trailing`.
    /// This operation preserves `isHidden`.
    public func setContentView(
        _ contentView: UIView,
        position: Position? = nil,
        animated: Bool = false
    )

    /// Mutates and remeasures installed content in one UIKit transaction.
    public func performContentUpdate(
        animated: Bool = true,
        _ updates: @escaping @MainActor () -> Void
    )

    /// Semantically detaches content before returning.
    public func removeContent(animated: Bool = false)

    /// Moves installed content, or records placement for the next render.
    public func setPosition(
        _ position: Position,
        animated: Bool = false
    )

    /// Changes requested visibility, including while content is absent.
    public func setHidden(
        _ hidden: Bool,
        animated: Bool = false
    )

    /// Changes the host tab bar visibility and coordinates accessory layout.
    public func setTabBarHidden(
        _ hidden: Bool,
        animated: Bool = false
    )
}
```

There is no public initializer, protocol, renderer, diagnostic hook, or test seam. `Position` is a closed-construction value with named static positions rather than a public enum, so a future named position can be added without breaking exhaustive consumer switches. Its internal normalized alignment is consumed by both renderers without platform-specific position switches.

Sizing contract: content must produce a fitting size through intrinsic content size, internal Auto Layout constraints, or nonzero bounds. FloatingAccessoryKit constrains its own host rather than installing required width/height constraints directly on consumer content. It does not assign a fixed minimum width to valid consumer content: standard controls such as `UIButton`, including buttons that present a `UIMenu`, contribute the fitting size proposed by UIKit. A consumer can make an image-only button square with a proportional `widthAnchor == heightAnchor` constraint; the platform accessory environment supplies the proposed height, so no fixed side length is needed. The content's fitting result determines its width, preserving wider natural sizes for titled or menu buttons. Actual host layout remeasures content automatically. Explicit changes to intrinsic size, bounds, constraints, or subview hierarchy go through `performContentUpdate(animated:_:)`, which executes the consumer mutation and remeasurement in one UIKit update transaction instead of polling or content resubmission. Initial measurement is applied immediately. Explicit updates animate by default with UIKit timing and respect Reduce Motion. On iOS 26, the transaction uses `UIView.AnimationOptions.flushUpdates` so UIKit flushes pending trait, property, and layout updates together for the native Liquid Glass presentation.

Alignment contract: `.leading` and `.trailing` pin the same logical edge of consumer content to the managed host while its width changes; `.center` keeps the centers aligned. This prevents unchanged edge controls from translating when the surrounding accessory grows or shrinks and preserves right-to-left behavior through logical anchors.

### 2.3 Consumer code

#### README / common case

Before:

```swift
private lazy var accessoryController = TabBarAccessoryController(
    tabBarController: self
)

accessoryController.setContent(
    button,
    position: .trailing,
    animated: false
)
```

After:

```swift
floatingAccessoryController.setContentView(button)
```

#### Sample / dynamic case

Before, the Sample stores a parallel position and hidden flag, calls `setContent` to remeasure, and reapplies position and visibility after content changes.

After:

```swift
let accessory = floatingAccessoryController
accessory.setContentView(button, position: selectedPosition, animated: true)
accessory.setHidden(shouldHideAccessory, animated: true)

// Later position change
accessory.setPosition(.center, animated: true)

// Later layout-affecting change
accessory.performContentUpdate {
    button.setTitle("Updated", for: .normal)
    button.invalidateIntrinsicContentSize()
}

// Removal; `button` may be reparented immediately after this returns.
accessory.removeContent(animated: true)
```

#### Proxy external package

The separate-package fixture imports `FloatingAccessoryKit`, obtains `floatingAccessoryController`, installs a button, changes placement and visibility, and removes it. Compilation proves that every step uses only shipped public declarations.

### 2.4 Access control plan

| Public declaration | Consumer reachability |
| --- | --- |
| `UITabBarController.floatingAccessoryController` | Entry point in all three consumer stories. |
| `TabBarAccessoryController` | Accessor result and receiver for all operations. |
| `TabBarAccessoryController.Position` | Placement parameter and Sample state type. |
| `Position.leading`, `.center`, `.trailing` | README/Sample placement choices. |
| `contentView` | Allows consumers to query current semantic attachment without mirror state. |
| `position` | Allows selection UI to read current placement without mirror state. |
| `isHidden` | Existing visibility query and Sample state synchronization. |
| `setContentView` | Install/replace story. |
| `performContentUpdate` | Coordinated mutation and dynamic sizing story when UIKit does not propagate host layout. |
| `removeContent` | Explicit removal and synchronous handoff story. |
| `setPosition` | Independent movement story. |
| `setHidden` | Independent requested-visibility story. |
| `setTabBarHidden` | iOS 18 tab-bar visibility and overlay-layout coordination. |

All renderers, host views, safe-area state, private-runtime helpers, animation state, logging, and composition functions remain internal.

### 2.5 Variation axes and addition tests

| Axis | Single absorption point | Variant-addition trace |
| --- | --- | --- |
| OS presentation | Renderer factory in the controller composition root | Add one renderer file and one factory registration; public controller and other renderers remain unchanged. |
| Horizontal placement | `Position`'s internal normalized horizontal alignment | Add one named static value in `TabBarAccessoryController.swift`; native and overlay renderers consume the normalized value without new switches. |

### 2.6 Deletion list

- Delete public `TabBarAccessoryController.init(tabBarController:)` and consumer-owned strong references. [F1, F3]
- Delete `setContent(_:position:animated:)` and its optional-`nil` removal route. [F2, F5]
- Delete the manual “call again with the same view” sizing contract and Sample callback that uses it. [F2, F7]
- Delete `TabBarAccessoryViewLifecycleHooks`, its global registry, four UIKit controller swizzles, suspension-depth bookkeeping, and corresponding tests. [F3, F6]
- Delete renderer-owned semantic `contentView`, `position`, and `isHidden` sources of truth; renderers retain only artifacts and transition state. [F2, F3]
- Delete safe-area baseline reconstruction by subtracting `appliedBottomInset`. [F4]
- Delete delayed ownership of the consumer view during animated removal; retain only a renderer-owned snapshot. [F5]
- Delete the Sample's mirror placement/visibility state where the controller can be queried directly. [F2, F3]
- Delete the DocC prohibition on consumer required size constraints and replace it with the single fitting-size contract. [F7]

### 2.7 Avoided shapes

- Do not keep `init(tabBarController:)` as deprecated SPI or add a registry that returns/coordinates separately created controller instances. The accessor is the only construction path.
- Do not make native and overlay renderers mutate `TabBarAccessoryController` state or carry their own semantic visibility/position mirrors.
- Do not replace `TabBarAccessoryViewLifecycleHooks` with another process-global `UITabBarController` hook registry. Geometry observation is installed per host/render tree, while tab-bar visibility changes use the explicit controller-owned coordination operation because UIKit exposes no public iOS 18 change notification.
- Do not preserve `setContent(nil, ...)` as an overload or deprecated shim. Removal remains a distinct operation.
- Do not poll arbitrary consumer views or use undocumented KVO to infer fitting-size changes. Host layout owns automatic measurement, and `performContentUpdate(animated:_:)` is the explicit boundary for consumer-driven changes.
- Do not start a separate consumer geometry animation before or after `performContentUpdate`; put the hierarchy, constraint, intrinsic-size, and alpha mutations in its closure so UIKit receives one transaction.
- Do not restore safe-area values by subtracting the last package delta from a composite current value.
- Do not animate the original consumer content after semantic removal; animate a snapshot or complete without animation.

### 2.9 Finding-to-design mapping

| Finding | Resolution |
| --- | --- |
| F1 duplicate controllers | Host-associated accessor, no public initializer, controller-lifetime tests. |
| F2 fused mutation and visibility reset | Explicit install/remove/move/hide/content-update operations with controller-owned state. |
| F3 duplicated/orphaned ownership and global hooks | Single semantic owner, renderer projections, per-host layout observation, deletion of lifecycle registry. |
| F4 unsafe safe-area reconstruction | Dedicated safe-area owner storing consumer baseline, contribution, and last package write separately. |
| F5 delayed content ownership | Synchronous semantic detach and snapshot-only animation. |
| F6 silent private hook failure | Capability-local fallback to public UIKit presentation plus one development diagnostic; global lifecycle swizzles removed. |
| F7 contradictory sizing contract | One fitting-size contract and host-owned constraints/measurement. |
| F8 no external contract build | Separate-package fixture and Sample build added to CI. |
| F9 split animation ownership and edge drift | Position-aware content-host anchors plus one controller-owned content update transaction using iOS 26 update flushing. |

## 3. README migration contract

The implementation phase will add this release section:

```markdown
## Migration

### v0.3.0

These notes apply when upgrading from `v0.2.x` or earlier to `v0.3.0`.

* Replace `TabBarAccessoryController(tabBarController:)` and the stored controller reference with the tab bar controller's `floatingAccessoryController`.
* Replace `setContent(view, position:animated:)` with `setContentView(view, position:animated:)`.
* Replace `setContent(nil, animated:)` with `removeContent(animated:)`.
* Use `setPosition(_:animated:)` for movement. Replace same-view resubmission for measurement with `performContentUpdate(animated:_:)`, placing every layout-affecting mutation in its closure; layout-driven changes remain automatic.
* `setContentView` and `removeContent` preserve requested visibility. `setHidden` also records visibility before content is installed.
* On iOS 18, replace direct `UITabBarController.setTabBarHidden(_:animated:)` calls with `floatingAccessoryController.setTabBarHidden(_:animated:)` so the overlay layout is updated in the same operation.
```
